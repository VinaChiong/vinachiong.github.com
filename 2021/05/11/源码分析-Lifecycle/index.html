1.概述

针对 Activity的‘生命周期’概念，抽象成具体的类型Lifecycle，接收生命周期事件，分发到各个‘观察者’实现。
‘生命周期’变得可观察、可感知，依赖‘生命周期’的工作，不再入侵到Activity、Fragment中。

‘接口隔离’、‘单一职责’：LifecycleObserver观察者不需要知道Activity、Fragment对生命周期的内部细节，只需要在对应Event来临时候处理好该干的活

2.Lifecycle中的运作流程及源码分析

Lifecycle组件的工作，是围绕「Event传递」展开：

LifecycleOwner -> Lifecycle -> LifecycleObserver

2.1 核心接口 LifecycleOwner

Activity 和 Fragment 都实现了 LifecycleOwner接口, 组件间的接口隔离，只依赖接口，不依赖具体类型

2.2 抽象类Lifecycle

  • 定义事件Event枚举类， LifecycleOwner 发出，以及LifecycleEventObserver接收
  • 定义自身状态State枚举类，根据接收的Event，进行变化：ordinal值升降，取值范围[0,4]
  • 抽象了LifecycleObserver的管理办法：允许观察者传入接口的实现接收Event

2.3 实现类 LifecycleRegistry

LifecycleOwner接口是由ComponentActivity实现，ComponentActivity默认创建一个LifecycleRegistry实例变量。

  • 弱引用持有 LifecycleOwner

LifecycleRegistry实例内部，以弱引用持有 LifecycleOwner（即Activity，Fragment）实例，保证Activity被GC回收时候，不会因Lifecycle或其内部类的强引用，导致错过回收，造成内存泄漏。

  • handleLifecycleEvent() 方法接收 Event

LifecycleRegistry通过 handleLifecycleEvent()方法，接收LifecycleOwner发送的Event：

  • 根据「映射规则」，变更自身State
  • 根据当前状态，把事件Event 「合适地」通知LifecycleObserver

源码Event枚举中，有静态方法：upTo, upFrom, downTo, downFrom

以下是 State变化时候，对应Event 的映射规则对照表：

借助对照表帮助我们理解4个方法：

​首先，方法的名称都修饰、作用于State：什么的Event导致State产生了变化
​up、down，表示“State值 加减 变化”, from表示该state值要变化，to表示成为这个state

​upTo(State) 表示: 获取「导致当前状态上升成入参State」的Event
​upFrom(State) 表示: 获取「导致入参State要加1」的Event
​downTo(State) 表示: 获取「导致当前状态下降成入参State」的Event
​downFrom(State) 表示: 获取「导致入参State要减1」的Event

  • LifecycleEventObserver的存储

LifecycleEventObserver的存储使用 FastSafeIterableMap<K, V>的数据结构，FastSafeIterableMap 的父类是 SafeIterableMap。

  • SafeIterableMap中是以Entry<K, V>类型链表的形式，存储数据，但API上是 map的形式。
  • FastSafeIterableMap在已有链表基础上，多添加HashMap进行保存。典型的空间换时间，能以K快速查找的同时，也可以使用迭代器做遍历操作

  • 静态内部类 ObserverWithState - LifecycleEventObserver的装饰者

    使用装饰者模式对LifecycleEventObserver进行“属性、行为扩展”的封装，记录 LifecycleEventObserver 对应 Lifecycle的状态State。为后面LifecycleRegistry的状态同步行为提供基础。

  • addObserver(LifecycleObserver obserber)方法：

    • 所有新增的Observer是添加到链表的尾部，并且马上与LifecycleRegistry.mState做同步。“同步”操作，保证observer能完整接收到所有已经发生的Event。
    • observer的默认状态 是 DESTROYED : INITIALIZED 二选一
    • 事件发送是通过 ObserverWithState.dispatchEvent(LifecycleOnwer, Lifecycle.Event)。
    • 当前新增observer完成同步后，再根据状态判断是否调用sync()方法对整个链表做检查并同步

- ReportFragment类

在ComponetActivity中，onCreate()方法中 调用ReportFragment.init(this)，添加一个ReportFragment实例到自身的Fragment事务管理。通过ReportFragment代为调用 LifecycleRegistry的 handleLifecycleEvent()

总结优势：

  • 模板方法模式 和 观察者模式
  • 避免在视图控件中注入监听代码

3.源码中的设计模式

  • 先说「观察者设计模式」 + 普通的回调

  • 放在Android Activity 中，MVP模式中也是常见的回调模式，但是需要自行判断View的状态，并且释放引用，这是弊端

  • 陈述：

    • 符合「迪米特原则（对象尽量对另一个对象少依赖）」，
    • LiveData+Lifecycle方案是一道横在「含生命周期组件 和 业务逻辑层」之间的接口隔离方案，
    • 业务层其实只想知道生命周期，而不是 Activity 和 Fragment
    • 一切交给LiveData，谁要观察LiveData的，就要遵循相关的约定：实现LiveData.Observer接口。

  • LiveData只知道Lifecycle，但也不知道Activity和Fragment的存在

  • 内部的GenerateAdapter，属于适配器模式
